# G1 Business Outcome Validator

Track post-launch business outcome validation for launched AWS opportunities.

## Goal: G1 - Validate Business Outcomes After Launch

Track tech team activities performed in 2025 that validate business outcomes after customer workload launches.

## Business Outcome Categories (2025)

All goals start with "G1 2025 - " prefix:

### Core Business Outcomes
- Improved Efficiency
- Generated Revenue
- Reduced Cost
- Improved Customer Satisfaction
- Mitigated Risk
- Launched Workload Discontinued
- Launched Workload Not Yet Delivered

### Public Sector (WWPS) Additional Outcomes
- Enhanced Public Value
- Improved Accessibility and Equity
- Strengthened Security and Resilience
- Advanced Policy Goals
- Promoted Environmental Stewardship
- Improved citizen/constituent experience

## Workflow

1. **Get Context**
   - Ask: "Account or Opportunity?"
   - If Account: search and select account
   - If Opportunity: search and select opportunity

2. **Verify Launched Status**
   - For opportunity-based: Check stage is "Launched" or "Completed"
   - Warn if opportunity is not launched
   - Allow user to proceed or cancel

3. **Business Outcome Details**
   - Ask for business outcome category (show numbered list)
   - Ask for validation date (default: today, must be post-launch)
   - Ask for 60-word business outcome context:
     - What was achieved after go-live?
     - Quantifiable metrics (e.g., "30% reduction", "$1M savings", "1,000 calls handled")
     - Business impact (efficiency, revenue, cost, satisfaction, etc.)
   - **Allow "skip" option:** User can type "skip" for basic auto-generated description

4. **Activity Type**
   - **Fixed SA Activity:** "Validation of Business Outcome after Launch [Management]"
   - No override needed - this is the only valid activity type for G1
   - No AWSI exception - all users must use this activity type

5. **Services and Domains**
   - Prompt for AWS services involved (optional, can extract from opportunity)
   - Prompt for technical domains (optional)
   - **Can be skipped:** User doesn't need to provide if not applicable

6. **Create Tech Activity**
   - **parentRecord**: opportunity ID or account ID
   - **activityDate**: validation date (must be >= launch date)
   - **status**: "Completed"
   - **subject**: "G1 Business Outcome Validation - {Customer}"
   - **description**: "G1 Business Outcome Context: {60-word summary}"
   - **saActivity**: "Validation of Business Outcome after Launch [Management]"
   - **goals**: ["G1 2025 - {selected outcome}"]
   - **services**: User-provided or extracted from opportunity
   - **domains**: User-provided or auto-suggested
   - **isVirtual**: true (default)

7. **Validation**
   - If opportunity-based, check:
     - ✅ Opportunity is launched (stage or probability = 100)
     - ✅ Valid estimated value (not $0, $1, or placeholder)
     - ✅ Activity date >= opportunity close date
     - Warn if missing but allow proceed

8. **Confirmation**
   - Show Salesforce URL for created activity
   - Show G1 goal tag applied
   - Offer to track another validation

## Description Guidelines

**User can type "skip" to auto-generate a basic description.**

**DO include (60-word context):**
- What was achieved after go-live
- Quantifiable metrics (percentages, dollar amounts, counts)
- Business impact (efficiency, revenue, cost, satisfaction)
- Timeframe (e.g., "after 3 months", "within first quarter")
- Customer quotes or feedback (if available)

**DO NOT include:**
- Pre-launch activities
- Implementation details
- Technical architecture
- Migration or setup processes

**Auto-generated description template (when user skips):**
```
G1 Business Outcome Context: After go live, the workload delivered measurable business value to {Customer}. The solution achieved {Outcome Category} through AWS services deployment. Post-launch validation confirmed successful business outcome realization.
```

**Good examples:**
- "After go live, the application handled 1,000 calls from customers. This reduced call center volume by 30%, resulting in $1M annual savings."
- "Migration completed Q4 2025. Customer realized 45% cost reduction by moving from on-premises to AWS, saving $500K annually in infrastructure costs."
- "New GenAI chatbot launched December 2025. Customer satisfaction scores increased from 3.2 to 4.5 stars. Support ticket volume decreased 40%."

**Bad examples (too vague):**
- "Customer is happy with AWS services."
- "Workload was successfully launched."
- "Migration went well."

## Validation Requirements

For opportunity-level activities:
- ✅ Opportunity stage is "Launched" or "Completed"
- ✅ Opportunity has valid estimated value (not $0, $1)
- ✅ Activity date >= opportunity close date (post-launch validation)
- ✅ Activity date is in 2025
- ✅ G1 goal tag applied
- ✅ Description includes 60-word business outcome context

## Date Validation

- **Must be in 2025:** G1 tracking is for 2025 activities
- **Must be >= opportunity close date:** Validation happens after launch
- **Default to today:** If not specified
- **Accept flexible formats:** "today", "2025-12-10", "early December"

If user provides date before opportunity close date:
```
⚠️ Warning: Validation date (2025-11-01) is before opportunity close date (2025-11-12).
   Business outcome validation should happen after workload launch.

   Use this date anyway? [Y/n]
```

## Implementation Notes

- Use aws-sentral MCP tools:
  - `search_opportunities` or `search_accounts` - find target
  - `get_opportunity_details` - verify opportunity info and launch status
  - `create_tech_activity` - create activity with goals
  - `get_opportunity_line_items` - verify products listed

- Goals parameter format:
  ```python
  goals = ["G1 2025 - Improved Efficiency"]
  ```

- Always set:
  - `saActivity`: "Validation of Business Outcome after Launch [Management]"
  - `status`: "Completed"
  - `isVirtual`: true (default)
  - `goals`: Array with one G1 2025 goal

- Subject line format:
  - "G1 Business Outcome Validation - {Customer Name}"

- Description format:
  - MUST start with "G1 Business Outcome Context: "
  - Include user-provided 60-word summary OR auto-generated if skipped
  - Keep focused on post-launch business outcomes

- Services extraction:
  - Try to extract from opportunity line items
  - If not available, ask user
  - Common services: Bedrock, SageMaker, S3, Lambda, etc.

- Domain auto-suggestion based on services:
  - Bedrock/SageMaker → "GenAI" or "Machine Learning"
  - Migration-related → "Migration & Modernization"
  - Cost optimization → "Cloud Operations"

   - AH3: AGS Canada
   - AH4/AH5: Specific to division/territory
   - Reporting Month: Current Month

### Step 2: Prioritize Older Opportunities
- On "Opportunity Details" tab, sort by close date (oldest to newest)
- Focus on validating older launched opportunities first

## Error Handling

- If opportunity/account not found: ask for different search term
- If opportunity not launched: warn and ask to proceed or cancel
- If validation date before close date: warn and ask to proceed
- If invalid date: prompt for correct format
- If description too short: prompt for more detail (60 words minimum)
- If missing products on opp: warn but allow proceed
- If activity creation fails: show error and offer retry
